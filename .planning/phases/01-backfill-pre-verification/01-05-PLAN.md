### Planner decision: flag semantics

Per CONTEXT.md D-14 and RESEARCH Open Question #4, this plan locks:
- Flag name: `--verify-concurrency`
- Default value: `0`
- `0` means auto-detect (`min(GOMAXPROCS, 4)`) — handled inside `verify.Verifier.Run` (Plan 01-01).
- `1` means strict serial.
- Negative values: clamped to 0 by `verify.WithConcurrency` (Plan 01-01). Kingpin allows negative ints through; the clamp ensures safe behavior.

The Action handler maps:
- `--skip-chunk-verification` true → `verify.Medium`, else `verify.Deep`.
- `--full-report` true → `WithFailFast(false)`, else `WithFailFast(true)` (default).
- `--verify-concurrency N` → `WithConcurrency(N)`.

### Mutual-exclusion layering (where to put the check)

Per the checker review (WARNING 7), the layering preference for the `--full-report` + `--skip-chunk-verification` mutual-exclusion check is:

1. **Primary: `cmd.Validate(...)` at the CmdClause level.** Researcher-verified kingpin API; fires after flags parse, before Action. Returns a clean error that kingpin formats as a flag-parse error (exit non-zero, no usage dump by default).
2. **Fallback A: `app.Validate(...)` at the Application level.** Runs after all parsing across all subcommands. Use only if `cmd.Validate` proves broken in the vendored kingpin version.
3. **Fallback B: `cmd.PreAction(...)`.** Runs before `cmd.Action`. Still cleaner than an Action-body check because kingpin treats PreAction errors as parse-time errors.
4. **DO NOT use a check inside the Action handler (`c.backfill()`).** Action errors trigger kingpin's Usage printout, producing a jarring, verbose error page for what should be a crisp one-line parse error. This ranking is a revision from earlier drafts that treated an Action-body check as an acceptable fallback — it is not.

Document in the plan SUMMARY which layer ended up holding the check so the test can target the right surface.

    t.Helper()
